# Prompt 迭代循环

> 这是一个元文档——它描述的流程也适用于它自己。
> 读完不理解没关系，跑完一轮再看。

---

## 核心循环

```
1. 应用里有一块逻辑需要 LLM + 提示词
2. 把提示词从应用代码中抽出来，放到一个 .md 里
3. 应用不再内置提示词，而是：读 .md → 动态生成提示词 → 调用 LLM
4. 运行应用，记日志（输入 / prompt / 输出 / 效果）
5. 分析日志 → 调整 .md → 期望下次生成更好的提示词
6. 重复 4-5，直到满意
```

## 在本项目中的应用

当前项目是 AI-Sale Demo，9 个场景各有自己的"AI 回复逻辑"。

| 层 | 对应什么 | 当前状态 |
|----|---------|---------|
| 应用 | scene-1 ~ scene-9 的 HTML + JS | 每个场景独立一套规则引擎 |
| LLM 模块 + 提示词 | 各场景的 `findAnswer()` / 话术生成逻辑 | 硬编码关键词匹配（不是 LLM） |
| 抽出来的 .md | `smart_mapping.md` | 已创建，但无人读取 |
| 日志 | `smart_mapping_log.md` | 已创建，记录了两轮观察 |
| 调 .md → 再跑 | — | 还没做 |

## 每次做事前的标准动作

```
1. 先读 smart_mapping.md（项目规则、数据结构、已知陷阱）
2. 再读 smart_mapping_log.md（上次踩了哪些坑）
3. 然后做具体的事
4. 做完后，把新发现记到 smart_mapping_log.md
5. 如果某个模式重复出现 → 升格到 smart_mapping.md
```

## 什么时候算"满意"

没有固定标准。以下信号出现任意一条就可以停一轮：

- 连续 2 次迭代没有新发现（日志只有"同上"）
- 当前场景的运行效果达到了 docs/OPTIMIZATION.md 里标的期望指标
- 日志里已经没有"未修复"的已知问题
- 你决定先做别的事

## 这个文档本身怎么迭代

这个文件描述的流程如果跑下来发现有问题（比如"先读 smart_mapping.md 再干活"这一步在实际中总是跳过），那就改这个文件本身。
