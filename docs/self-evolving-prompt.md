# 自进化提示词机制

## 核心理念

提示词不应该是写死的文本，而应该是一个**运行时由 LLM 自己生成的产物**。

应用里只存"元信息"（规则、数据描述、上下文），LLM 每次调用时读这些元信息，自己组装出最终的提示词。然后跑完看效果，调元信息，下一轮生成更好的提示词。

## 6 步循环

```
① 应用里有 LLM 模块，也有提示词

② 把提示词从应用代码中抽出来，放到一个 .md 里
   → 应用不再内置提示词
   → .md 里存的是"元信息"，不是最终提示词

③ LLM 读 .md 中的元信息，动态生成提示词
   → 最终提示词第一次出现是在 LLM 的推理中，不是在代码里

④ 跑应用，记日志
   → 记：用户输入 / LLM 读到的元信息 / LLM 生成的提示词 / 输出 / 效果

⑤ 分析日志 → 调 .md
   → 哪些元信息导致了好的输出？哪些导致了差的？
   → 调规则、调上下文、调数据引用方式

⑥ 重复 ④-⑤，直到满意
```

## 关键区别

| | 传统方式 | 这个机制 |
|---|---|---|
| 提示词存在哪 | 代码里写死 | LLM 运行时生成 |
| .md 里有什么 | 最终提示词 | 元信息（规则/上下文/数据描述） |
| 谁读 .md | 开发者（写代码时参考） | LLM（运行时读取） |
| 迭代主体 | 开发者手动改 | LLM + 日志驱动 |
| "动态"的含义 | 变量替换（模板 + 参数） | 提示词本身由 LLM 产出 |

## 适用范围

这套机制适合的场景特征：

- **提示词频繁变**：规则、上下文、数据引用方式经常需要调
- **输出质量依赖上下文**：同样的逻辑在不同项目/场景下需要不同的表达
- **有日志可分析**：能记录输入→输出→效果，形成反馈

不适合的场景：

- 提示词极简单且从不变化（写死更快）
- 没有日志/无法判断输出好坏
- 延迟敏感（LLM 生成提示词多一次调用）
