# PRD：产品化 SOP 系统

> **版本**: v3.0（方向修正终版） | **日期**: 2026-05-23
> **核心转变**: 从"系统硬编码规则"到"运营人员在界面上配置人群+规则，引擎按配置执行"

---

## 0. 产品定义

**一句话**：一个让运营人员**自己配置**「**对谁、在什么时候、做什么**」的自动化客户跟进系统。

### 为什么必须可配置？

```
一个实际的销售团队在使用 SOP 系统时：

第一周：
  运营："新客户第一天发欢迎消息"
  运营："第三天发案例"
  运营："第七天约试菜"
  → 配置进系统

第二周：
  运营："连锁客户和食堂客户的欢迎词不一样"
  → 创建两个人群，配不同的欢迎规则

第三周：
  运营："连锁客户试菜后三天没动静的要报异常给主管"
  → 追加一条规则

一个月后：
  团队跑顺了，沉淀了 8 个人群、20 条规则
  → 这是团队的 SOP 资产，存在系统里，不是写在代码里
```

**如果不配置化**，每次调整都要改代码——这不是产品，是外包项目。

## 1. 三层架构

```
┌──────────────────────────────────────────────────────────────┐
│  ① 人群管理 (Audience)                                       │
│  作用：定义"对谁执行"                                        │
│  输入：条件表达式（type=CHAIN AND budget>10万...）            │
│  输出：符合条件的客户列表                                     │
│  界面：人群列表 + 条件编辑器 + 匹配预览                       │
├──────────────────────────────────────────────────────────────┤
│  ② 规则配置 (Rule)                                           │
│  作用：定义"什么时候 + 做什么"                                │
│  输入：选定人群 + 触发条件 + 前置条件 + 执行动作              │
│  输出：一条可执行的规则                                       │
│  界面：规则列表 + 规则编辑器                                  │
├──────────────────────────────────────────────────────────────┤
│  ③ 执行引擎 (Executor)                                       │
│  作用：执行已配置的规则                                      │
│  输入：规则配置 + 客户数据                                    │
│  输出：消息发送/阶段迁移/异常上报/执行日志                    │
│  界面：执行监控 + 异常处理中心 + 执行日志                    │
└──────────────────────────────────────────────────────────────┘
```

## 2. 人群管理（Audience）

### 2.1 人群模型

```json
{
  "audienceId": "aud-001",
  "name": "连锁餐饮新客户",
  "description": "刚进入P1的连锁品牌客户",
  "conditions": {
    "logic": "AND",
    "rules": [
      { "field": "typeCode", "op": "IN", "value": ["CHAIN", "HOTEL"] },
      { "field": "stage", "op": "EQ", "value": "P1" },
      { "field": "budget", "op": "GTE", "value": 100000 }
    ]
  },
  "enabled": true,
  "createdAt": "2026-05-23",
  "matchCount": 0        // 每次扫描后更新
}
```

### 2.2 条件操作符

| 操作符 | 含义 | 适用字段 |
|--------|------|---------|
| EQ | 等于 | stage, status, typeCode |
| IN | 属于集合 | typeCode, source |
| GTE | 大于等于 | budget（数值化后） |
| LTE | 小于等于 | budget |
| DAYS_GT | 距今超过N天 | lastContact, stageEnteredAt |
| CONTAINS | 包含 | tags |
| MATCHES | 关键词匹配 | name, company |

### 2.3 条件组合逻辑

支持 AND / OR 嵌套组合：

```json
{
  "logic": "AND",
  "rules": [
    { "field": "stage", "op": "EQ", "value": "P1" },
    {
      "logic": "OR",
      "rules": [
        { "field": "typeCode", "op": "EQ", "value": "CHAIN" },
        { "field": "typeCode", "op": "EQ", "value": "CANTEEN" }
      ]
    }
  ]
}
```

## 3. 规则配置（Rule）

### 3.1 规则模型

```json
{
  "ruleId": "rule-001",
  "name": "连锁新客Day1欢迎",
  "enabled": true,
  "audienceId": "aud-001",
  "trigger": {
    "type": "stage_entry",           // stage_entry | stage_entry_after | keyword_match | time_elapsed
    "stage": "P1",
    "delayDays": 0                   // 0=立即, 3=进入后3天
  },
  "conditions": {
    "maxMessagesToday": 2,           // 频率限制
    "requireNoReply": true,          // 仅当客户未回复时
    "requireStage": "P1"             // 仅当客户仍在P1
  },
  "action": {
    "type": "send_message",          // send_message | stage_transition | update_tag | escalate
    "channel": "wecom",
    "contentTemplate": "您好{customerName}！我是优特智厨的销售顾问……"
  },
  "escalation": null,                // 选填：执行后上报给销售
  "stats": {
    "totalExecuted": 0,
    "lastExecutedAt": null
  }
}
```

### 3.2 触发类型

| 触发类型 | 含义 | 示例 |
|---------|------|------|
| stage_entry | 进入某阶段时立即执行 | 客户进入P1 → 发欢迎消息 |
| stage_entry_after | 进入某阶段N天后执行 | 进入P1后3天 → 发方案 |
| keyword_match | 客户回复含关键词时 | 客户说"试菜" → P1→P2 |
| time_elapsed | 超过N天无互动时 | 30天无互动 → 标记沉睡 |

### 3.3 动作类型

| 动作类型 | 含义 | 参数 |
|---------|------|------|
| send_message | 发送消息 | channel + contentTemplate |
| stage_transition | 迁移阶段 | fromStage? + toStage |
| update_tag | 更新标签 | tags (add/remove) |
| escalate | 上报异常 | level + message + assignTo |

## 4. 执行引擎

```
每日定时扫描（或手动触发）
  ↓
1. 遍历所有启用的「人群」
   ↓
2. 对每个人群，执行条件匹配 → 得到当前客户列表
   ↓
3. 对每个匹配客户，遍历该人群下的所有「规则」
   ↓
4. 检查规则的触发条件
   ├─ stage_entry：客户刚进入指定阶段？
   ├─ stage_entry_after：进入指定阶段是否已达N天？
   ├─ keyword_match：客户的新回复中是否含关键词？
   └─ time_elapsed：是否超过N天无互动？
   ↓
5. 满足触发条件 → 检查前置条件
   ├─ maxMessagesToday？ → 今天已发未超限
   ├─ requireNoReply？ → 客户未回复
   └─ requireStage？ → 客户仍在指定阶段
   ↓
6. 所有条件满足 → 执行动作
   ├─ send_message → 发送+记录
   ├─ stage_transition → 更新阶段
   ├─ update_tag → 更新标签
   └─ escalate → 创建异常
   ↓
7. 记录执行日志
```

## 5. 数据模型总表

### 5.1 人群（Audience）

| 字段 | 类型 | 说明 |
|------|------|------|
| audienceId | string | 唯一标识 |
| name | string | 人群名称 |
| description | string | 描述 |
| conditions | ConditionGroup | 条件表达式树 |
| enabled | boolean | 是否启用 |
| createdAt | string | 创建日期 |
| updatedAt | string | 更新日期 |
| matchCount | number | 最近匹配客户数 |

### 5.2 规则（Rule）

| 字段 | 类型 | 说明 |
|------|------|------|
| ruleId | string | 唯一标识 |
| name | string | 规则名称 |
| enabled | boolean | 是否启用 |
| audienceId | string | 关联人群ID |
| trigger | Trigger | 触发条件 |
| conditions | RuleConditions | 前置条件 |
| action | Action | 执行动作 |
| escalation | Escalation? | 异常上报配置 |
| createdAt | string | 创建日期 |
| stats | RuleStats | 执行统计 |

### 5.3 执行日志（ExecutionLog）

| 字段 | 类型 | 说明 |
|------|------|------|
| logId | string | 唯一标识 |
| executedAt | string | 执行时间 |
| customerId | string | 相关客户 |
| ruleId | string | 执行的规则 |
| action | string | 执行的动作类型 |
| result | string | 执行结果 |
| detail | string | 详情（发送内容/迁移方向等） |
